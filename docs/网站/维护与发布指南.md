# 维护与发布指南

这份文档给出本站日常维护的标准流程：修改文字、发布新文章、同步到 GitHub 并上线。

## 为什么放在这里

本说明放在 `网站` 栏目最合适，因为它属于站点运维文档，而不是业务内容文章。后续你或其他人接手时，也能在同一位置快速找到。

## 一次完整流程

1. 修改或新增 `docs/` 下的 Markdown 文件。
2. 本地预览确认效果。
3. 提交并推送到 GitHub。
4. 等待 GitHub Actions 自动部署到 Pages。

## 如何修改网站上的文字

1. 找到对应文件，例如：
   - 首页：`docs/index.md`
   - 关于我：`docs/我/about.md`
   - 网站说明：`docs/网站/*.md`
2. 直接编辑文字并保存。
3. 本地预览：

```bash
cd ~/my-blog
mkdocs serve
```

4. 打开 `http://127.0.0.1:8000` 检查页面。

## 如何发布一篇新文章

以发布到“博客”栏目为例。

1. 新建文章文件：`docs/博客/文章标题.md`
2. 在文件内写内容，例如：

```md
# 文章标题

这是正文。
```

3. 把文章加入导航 `mkdocs.yml`：

```yaml
- 博客:
    - 概览: 博客/index.md
    - 文章标题: 博客/文章标题.md
```

4. 本地预览确认无误。

## 如何更新 GitHub 并上线

```bash
cd ~/my-blog
git add .
git commit -m "update docs"
git push
```

推送后会自动触发：`Deploy MkDocs to GitHub Pages`。

## 如何确认发布成功

1. 打开 GitHub 仓库的 `Actions` 页面。
2. 查看最新工作流 `Deploy MkDocs to GitHub Pages` 是否为绿色成功。
3. 打开线上地址：`https://huang-huaichao.github.io/`
4. 如果页面未更新，等待 1-3 分钟并强制刷新浏览器缓存。

## 常用命令速查

```bash
# 本地预览
mkdocs serve

# 严格构建检查
mkdocs build --strict

# 提交并发布
git add . && git commit -m "update docs" && git push
```

## 常见问题

### Q1: 改了文件但页面没变

- 先确认改的是 `docs/` 下文件。
- 再确认 `mkdocs.yml` 是否已加入对应导航。

### Q2: 推送后线上没更新

- 先看 Actions 是否成功。
- 再确认仓库 `Settings -> Pages -> Source` 为 `GitHub Actions`。

### Q3: 本地端口被占用

```bash
mkdocs serve -a 127.0.0.1:8001
```
